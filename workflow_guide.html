chunk_text() — how splitting works

# sliding window chunker
text = "...full document text..."
chunk_size = 800    # characters
overlap    = 120    # overlap between chunks
step = chunk_size - overlap   # = 680

chunks = []
for i in range(0, len(text), step):
    chunk = text[i : i + chunk_size]
    if len(chunk) > 50:    # skip tiny tail
        chunks.append(chunk)

# Each chunk stored as Chunk ORM row:
# chunk_index=0,1,2,...  text=...  vector_id=None

Section 04 · Workflow C

RAG Indexing — Chunks → Embeddings → ChromaDB

Fires when you click "Index & Apply" on the RAG Config screen

RagService.index() — full method trace

async def index(self, payload: RagIndexRequest):
    # 1. Load all SQLite chunks for this use case
    chunks = db.query(Chunk).filter(Chunk.usecase_id == payload.usecase_id).all()
    if not chunks: return RagIndexResponse(status="no_chunks")

    # 2. Get or create ChromaDB collection for this use case
    collection = chromadb.PersistentClient(path=vector_store_dir)
                         .get_or_create_collection(f"uc_{usecase_id}",
                           metadata={"hnsw:space": "cosine"})

    # 3. Get embedding provider (openai or ollama)
    embedder = EmbeddingProvider(payload.embedding_provider)

    # 4. Process in batches of 50
    for batch in batches(chunks, size=50):
        texts      = [c.text for c in batch]
        ids        = [c.id   for c in batch]
        metadatas  = [{"filename": ..., "chunk_index": ...} ...]

        # 5. Generate embeddings (async)
        embeddings = await embedder.embed_batch(texts)
        # Ollama: POST /api/embeddings ×N  |  OpenAI: client.embeddings.create(input=texts)

        # 6. Upsert into ChromaDB
        collection.add(embeddings=embeddings, documents=texts,
                       metadatas=metadatas, ids=ids)

        # 7. Write vector_id back to SQLite chunk row
        for chunk in batch: chunk.vector_id = chunk.id

    # 8. Mark documents as indexed
    db.query(Document).filter(id.in_(doc_ids)).update({"status": "indexed"})
    db.commit()
    return RagIndexResponse(indexed_chunks=indexed, status="indexed")
Ollama Path

nomic-embed-text

Each text sent individually via httpx async POST to
localhost:11434/api/embeddings
Returns 768-dimensional float vector.
No API key. No cost. Runs locally.
OpenAI Path

text-embedding-3-small

Full batch sent in one API call via
AsyncOpenAI.embeddings.create(input=texts)
Returns 1536-dimensional float vectors.
Requires valid OPENAI_API_KEY.
Critical Rule

Provider Must Match

You must use the same embedding provider at index time and query time.
768-dim vectors cannot be compared to 1536-dim vectors — ChromaDB will error.
Provider saved in rag_configs.embedding_provider.
Section 05 · Workflow D — The Core

6-Node Agent Pipeline — POST /agent/ask

Every LLM call, retrieval step and decision that fires when you hit "Ask"

N1
Intent Analyzer
LLM Call #1
Input: raw question + use case name
Prompt: "Analyze this question. Return JSON with intent label, question_type, requires_citation, risk_sensitive"
Output written to: state.intent, state.question_type, state.risk_sensitive
_node_intent_analyzer(state, provider) 
// LLM returns JSON like:
{
  "intent": "damaged_delivery_inquiry",
  "question_type": "procedural",
  "requires_citation": true,
  "risk_sensitive": false
}
N2
Retriever
Vector Search — NO LLM
Input: question text + top_k from rag config
Step 1: fast-path check — if ChromaDB collection is empty → return [] immediately
Step 2: embed question using saved rag_config.embedding_provider
Step 3: collection.query(query_embedding, n_results=top_k)
Output: list of ChunkResult with score, filename, text_preview
RagService.retrieve_for_agent() 
// Returned chunk example:
{
  "chunk_id": "3dc6927a-...",
  "filename": "shopeasy_policy.txt",
  "score": 0.8741,
  "text_preview": "If you receive a
  damaged or wrong item,
  contact support@..."
}
N3
Answer Generator
LLM Call #2 — Most Important
If no chunks retrieved: sets draft = "I could not find enough context…" and skips.
System prompt: includes answer_style instruction (concise / detailed / support-agent / legal-review / technical-runbook)
User prompt: formatted context blocks [1] filename (score: 0.87)\n{text} + question
Output: state.draft_answer, state.citations[]
_node_answer_generator(state, provider) 
// Context injected as:
[1] shopeasy_policy.txt (0.87)
If you receive a damaged
or wrong item, contact...

[2] shopeasy_policy.txt (0.81)
Refund methods: original
payment method: full...
N4
Grounding Verifier
LLM Call #3 — Quality Gate
Checks whether the draft answer is actually supported by the retrieved context (faithfulness).
Uses only the first 3 chunks + first 500 chars of the answer to keep tokens low.
Returns a 0–1 score. If it fails/times out, estimates 0.75 so the pipeline continues. 
// LLM prompt output:
{ "faithfulness": 0.92, "unsupported_claims": [], "status": "passed" }
_node_grounding_verifier(state, provider)
N5
Risk Classifier
LLM Call #4 — Safety Gate
Classifies the risk of a user acting on this answer:
low — informational, read-only (What is the return policy?)
medium — operational or business decision (Should we refund this customer?)
high — legal, financial, destructive, privacy (How do I terminate this employee?) 
{ "risk_level": "low", "approval_required": false, "risk_reason": "Read-only policy question" }
_node_risk_classifier(state, provider)
N6
Final Responder
NO LLM — Pure Logic
Copies draft_answer to final_answer. If approval_required=true and risk is high, appends an escalation warning banner to the bottom of the answer.
Then calls _persist_run() to write the full run to SQLite: agent_runs + agent_trace_steps + retrieved_chunks.
_node_final_responder(state) _persist_run(state, provider)
